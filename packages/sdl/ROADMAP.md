# @arch0/sdl Package Roadmap

This document tracks planned implementation work for the `@arch0/sdl` reference package. It covers generator wiring, CLI behaviour, test coverage, and tooling — not language spec decisions.

For SDL language evolution (new sections, version semantics, migration policy), see [`spec/ROADMAP.md`](../../spec/ROADMAP.md).

---

## Current State

- 480 tests, 0 failures
- 12 registry-backed artifact types, 5 direct API generators
- All outputs carry `tier: deterministic | inferred | advisory`
- `compile()` returns `inferences[]` — every normalizer-applied default is visible
- Core pipeline (parse → validate → normalize → resolve → generate) is production-quality

Gaps:
- `contracts` and `domain` are not read by any generator despite having enforced schemas
- Advisory generator outputs have no in-file review signal
- CLI does not surface generator tiers to end users
- Progress tracker, ADR draft generator, diff, and pricing have thin test coverage
- No `sdl migrate` tooling for documents authored against older shapes

---

## Item 1 — Wire `contracts` into `openapi`

**What:** Update `generateOpenApiSpec()` to read `contracts.apis[]` as the authoritative API surface when present, rather than inferring solely from `architecture.services` and `product.personas`.

**Behaviour:**
- If `contracts.apis[]` is present: use it as the canonical list of API contracts; derive server definitions, security schemes, and tag groupings from it
- If absent: current inference behaviour (from architecture + product) continues unchanged — no breaking change
- `x-version`, `x-basePath`, `x-endpoints` extension fields on `contracts.apis[]` items inform the generated spec when present

**Tests:** fixtures with and without `contracts` section; assert that `contracts`-informed output differs from inferred output in expected ways; assert backward compatibility when `contracts` is absent

**Section support change:** `contracts` moves from `minimal` to `partial` in `reference/section-support.md`

---

## Item 2 — Wire `domain` into `data-model`

**What:** Update `generateDataModel()` to read `domain.entities[]` as the authoritative entity definitions when present, rather than inferring from `product.personas` and `product.coreFlows`.

**Behaviour:**
- If `domain.entities[]` is present: generate ORM schema directly from entity field definitions (`name`, `type`, `nullable`, `primaryKey`, `foreignKey`, `unique`, `generated`, `default`, `enum`, `maxLength`, etc.)
- If absent: current inference behaviour continues unchanged
- Relationships in `domain.relationships[]` inform foreign key and join table generation

**Tests:** fixtures with rich `domain` sections; assert that field-level ORM output matches domain definition; assert backward compatibility

**Section support change:** `domain` moves from `minimal` to `partial`

---

## Item 3 — Wire `slos` into `monitoring`

**What:** Update `generateMonitoring()` to read `slos.services[]` for alert threshold configuration when present.

**Behaviour:**
- `slos.services[].availability` drives the Prometheus availability alert threshold for that service
- `slos.services[].latencyP95` drives the histogram quantile alert threshold
- If `slos` is absent: current hardcoded defaults continue

**Tests:** fixtures with `slos` section; assert alert thresholds match declared SLOs

**Section support change:** `slos` moves from `minimal` to `partial`

---

## Item 4 — Wire `resilience` into `coding-rules`

**What:** Update `generateCodingRules()` to emit resilience-specific rules when `resilience` is present.

**Behaviour:**
- `resilience.circuitBreaker.enabled: true` → add circuit breaker pattern rules (never call downstream without breaker, use established library per framework)
- `resilience.retryPolicy` → add retry rules (cap retries, use exponential backoff, make retried ops idempotent)
- `resilience.timeout.default` → add timeout rules (all outbound calls must have timeout set)
- `resilience.rateLimit` → add rate limit enforcement rules

**Tests:** fixtures with and without resilience section

**Section support change:** `resilience` moves from `placeholder` to `partial` (pending spec shape confirmation from `spec/ROADMAP.md`)

## Item 5 — Advisory output review headers

**What:** Each file generated by an advisory generator (`adr`, `backlog`, `deployment-guide`, `cost-estimate`) should open with a comment block identifying it as a generated draft requiring review.

**Format (Markdown files):**
```markdown
<!-- sdl:generated tier:advisory — review and edit before committing -->
```

**Format (YAML/config files):** not applicable — advisory generators currently produce Markdown only

**Why:** The `tier` field exists in the API response but does not survive outside the session. A developer who opens a generated ADR file a week later has no signal that it is a draft.

**Tests:** assert header presence in advisory generator output; assert header absent in deterministic output

## Item 6 — CLI tier summary on generation

**What:** When generating via CLI, print a tier summary after generation completes.

**Format:**
```
Generated 8 artifacts:

  deterministic (4)  architecture-diagram, repo-scaffold, openapi, data-model
  inferred (2)       coding-rules, coding-rules-enforcement  ← review before committing
  advisory (2)       adr, backlog  ← drafts, edit before use

Advisory outputs require review. Run with --tier deterministic to skip them.
```

**`--tier` flag:** accept `deterministic`, `inferred`, `advisory`, or `all` (default). Restricts which tiers are generated. Useful in CI to skip advisory outputs.

## Item 7 — `sdl migrate` command

**What:** CLI command that reads an SDL document, detects stale or invalid patterns, and outputs a corrected document with a diff of what changed.

**Migration cases (from canonical-contract.md rejected forms):**
- `stage: mvp` → `stage: MVP`
- `stage: growth` → `stage: Growth`
- `stage: enterprise` → `stage: Enterprise`
- `framework: express` → `framework: nodejs`
- `framework: next` → `framework: nextjs`
- `framework: fastapi` → `framework: python-fastapi`
- `auth.strategy: jwt` → `auth.strategy: oidc` + `auth.sessions.accessToken: jwt`
- `contracts: [...]` → `contracts: { apis: [...] }`
- `features: { phase1: [...] }` → `features: [...]` (flattened with `x-phase` annotation)
- `slos: [...]` → `slos: { services: [...] }`
- Custom root keys without `x-` → prefixed with `x-`

**Output:** prints migrated YAML to stdout; `--in-place` flag rewrites the file; `--diff` flag shows only the changes

**Tests:** one fixture per migration case asserting input → output; round-trip test asserting migrated output compiles clean

## Item 8 — Test coverage parity for exported APIs

**What:** Every exported function has at least one dedicated test file with meaningful fixtures.

**Current gaps:**

| Export | Coverage gap |
|---|---|
| `resolveProgress()` | No fixtures for partial, complete, or over-specified codebases |
| `generateAdrDraftFromDiff()` | No fixtures asserting which diffs trigger ADR generation |
| `classifyDiffForAdr()` | No fixtures for each `AdrCategory` |
| `diff()` | Missing edge cases: empty document, section added/removed, nested field changed |
| `buildPricingSummaryForPrompt()` | No fixtures asserting cost ranges for known cloud/stack combos |
| `generateDeployDiagram()` | Only exported, not covered in generators test suite |

**Exit criteria:** `npm test` catches regressions in all exported API surface, not just the core compile pipeline

## Item 9 — Compliance checklist generator

**What:** A new advisory generator that reads `compliance.frameworks[]` and produces a Markdown compliance checklist per framework.

**Trigger:** `artifacts.generate` includes `compliance-checklist` (new `ArtifactType` value), or called directly via `generateComplianceChecklist()`

**Output:** per-framework checklist of implementation requirements derived from the named frameworks. Does not require a strict `compliance` schema — reads the `name` field and matches against a built-in framework knowledge base.

**Tier:** `advisory` — checklist is a starting point, not a compliance audit

**Why:** The compliance section currently drives nothing. This is the lowest-cost path to making it useful without requiring schema formalization first.
